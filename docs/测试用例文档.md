### 1.3 测试方法：Tomcat + Curl + 数据库验证

本项目采用 **命令行自动化测试** 方式，通过 curl 模拟 HTTP 请求，结合数据库查询验证结果，确保测试准确可靠。

#### 1.3.1 测试流程

```
1. 启动 Tomcat 服务器
2. 使用 curl 模拟登录（获取 Session Cookie）
3. 使用 curl 发送业务请求（GET/POST）
4. 使用 mysql 命令查询数据库验证结果
5. 检查 HTTP 响应状态码和重定向 URL
```

#### 1.3.2 常用命令示例

**登录获取 Session：**
```bash
# 管理员登录
curl -s -c cookies.txt -d "username=admin&password=admin123" \
  "http://localhost:8080/Software_group/login" -L -o /dev/null

# 成员登录
curl -s -c cookies_member.txt -d "username=member1&password=123456" \
  "http://localhost:8080/Software_group/login" -L -o /dev/null
```

**发送 GET 请求：**
```bash
curl -s -b cookies.txt \
  "http://localhost:8080/Software_group/activity?action=approve&activityId=6&userId=2"
```

**发送 POST 请求：**
```bash
curl -s -b cookies.txt -X POST \
  "http://localhost:8080/Software_group/activity?action=register&activityId=10"
```

**验证 HTTP 响应和重定向：**
```bash
curl -s -b cookies.txt \
  "http://localhost:8080/Software_group/activity?action=approve&activityId=6&userId=2" \
  -w "\nHTTP:%{http_code}\nREDIRECT:%{redirect_url}\n"
```

**查询数据库验证：**
```bash
# 查询报名状态
mysql -u root -p123456 -D software_group -e \
  "SELECT activity_id, user_id, status FROM activity_participant WHERE activity_id = 6;"

# 查询活动信息
mysql -u root -p123456 -D software_group -e \
  "SELECT id, name, registration_start_time, registration_end_time FROM activity WHERE id = 6;"
```

#### 1.3.3 验证要点

| 验证项 | 方法 | 预期 |
|--------|------|------|
| 功能执行 | HTTP 状态码 | 302（重定向）或 200 |
| 数据变更 | 数据库查询 | status 字段正确更新 |
| 权限控制 | 未登录请求 | 重定向到登录页 |
| 业务逻辑 | 数据库状态 | 如：过期活动不可审核 |

#### 1.3.4 重启 Tomcat

```bash
# 停止
taskkill //F //IM java.exe

# 启动（PowerShell）
Start-Process -FilePath 'D:\Program Files\tomcat\apache-tomcat-9.0.113\bin\startup.bat'
```

#### 1.3.5 重新加载代码

修改 Java 代码后需要重新编译并触发 Tomcat 重新加载：

```bash
# 1. 编译 Java 文件
javac -encoding UTF-8 -d "webapps/Software_group/WEB-INF/classes" \
  -cp "tomcat/lib/servlet-api.jar;tomcat/lib/jsp-api.jar;..." \
  src/servlet/ActivityServlet.java

# 2. 访问任意页面触发 reload
curl -s "http://localhost:8080/Software_group/activity?action=manage"
```

## 六、常见问题与解决方案

### 6.1 部署目录差异导致文件404

**问题现象**：
- 上传的头像文件在页面上显示404错误
- 数据库中file_storage表的记录存在，但文件无法访问

**原因**：
项目可能使用不同的部署方式，文件存储位置不同：
- **命令行启动Tomcat**：文件部署在 `Tomcat/webapps/Software_group/localstorage/`
- **IDEA运行**：文件部署在 `项目/out/artifacts/Software_group_v2/localstorage/`

**解决方案**：
1. 确保使用统一的部署方式进行开发和测试
2. 如果使用IDEA运行，检查IDEA的输出目录：`out/artifacts/Software_group_v2/`
3. 如果使用命令行Tomcat，检查webapps目录：`webapps/Software_group/`
4. 同步两个目录的文件，或修改代码中的文件存储路径为统一位置

**排查步骤**：
```bash
# 1. 检查数据库中file_storage表的file_path字段
mysql -u root -p123456 -D software_group -e "SELECT id, file_path FROM file_storage WHERE id = ?;"

# 2. 根据部署方式检查对应目录
# 命令行Tomcat
dir "D:\Program Files\tomcat\apache-tomcat-9.0.113\webapps\Software_group\localstorage\images\avatar"
# IDEA
dir "D:\Code\自动工具\opencode\Software_group\out\artifacts\Software_group_v2\localstorage\images\avatar"
```
