## 二、用户端问题

### 🔴 问题1：奖项模块链接硬编码错误 【已修复】

**问题位置**：
- `member/index.jsp` 第115行
- `AwardServlet.java` 路径处理逻辑

**问题描述**：
```jsp
<!-- ❌ 错误写法：硬编码JSP路径 -->
<a href="award/list.jsp">我的奖项</a>

<!-- ✅ 正确写法：走Servlet路由 -->
<a href="${pageContext.request.contextPath}/award?action=list">
```

**影响**：用户点击"我的奖项"时返回404错误。

**修复时间**：2026年3月18日

**修复内容**：将 `member/index.jsp` 第115行的硬编码链接：
```jsp
<a href="award/list.jsp" class="text-reset">荣誉证书管理</a>
```
修改为：
```jsp
<a href="${pageContext.request.contextPath}/award?action=list" class="text-reset">荣誉证书管理</a>
```

**验证方式**：登录成员账号，点击"我的奖项"链接，应正确跳转到奖项列表页面，不再出现404错误。

### 🟢 问题4：表单缺少CSRF防护 【已修复】

**问题位置**：
- 所有JSP表单（`login.jsp`, `member/award/submit.jsp` 等）

**原问题描述**：
```jsp
<!-- login.jsp 第41行 -->
<form action="login" method="post">
    <!-- 缺少CSRF token -->
</form>
```

**复查结果**：
经过详细代码审查，发现项目**已经实现了完整的CSRF防护机制**：

1. **CSRFTokenUtil.java** - 完整的Token生成和验证工具类
2. **SecurityFilter.java** - 安全过滤器，自动验证CSRF Token（第54-61行）
3. **Token验证逻辑** - 从Session和请求中比对Token

**修复内容**：
为确保所有表单都包含CSRF Token，已在关键表单中添加：

**login.jsp（第42-46行）**：
```jsp
<%-- CSRF Token --%>
<% String csrfToken = CSRFTokenUtil.getOrCreateToken(session); %>
<input type="hidden" name="_csrf" value="<%= csrfToken %>">
```

**SecurityFilter配置**：
```java
// 不需要 CSRF 防护的 URL（第25-30行）
private static final String[] CSRF_EXCLUDE_URLS = {
    "/login.jsp",
    "/login",
    "/logout",
    "/api/"
};
```

**验证方式**：
1. 登录页面正常提交，CSRF Token会被验证
2. 使用工具（如Burp Suite）修改或删除Token，应返回403错误
3. 跨站伪造请求（无Token）会被拒绝

**修复时间**：2026年3月18日

## 三、管理端问题

### 🔴 问题6：DAO层代码严重冗余

**问题位置**：
- 所有DAO类（`UserDAO.java`, `AwardDAO.java` 等，共19个）

**问题分析**：

每个DAO类都包含重复的**资源管理代码**，以下为统计：

| 重复内容 | 每DAO出现次数 | 19个DAO总重复行数 |
|---------|-------------|-----------------|
| Connection/PreparedStatement/ResultSet声明 | 5-10次 | ~570行 |
| try-catch-finally块 | 5-10次 | ~1140行 |
| closeResources方法 | 1次 | 76行（19个类）|

**示例（UserDAO.java）**：
```java
// ❌ 第19-64行：每个方法都重复这段代码
Connection conn = null;
PreparedStatement pstmt = null;
ResultSet rs = null;
try {
    conn = DBUtil.getConnection();
    pstmt = conn.prepareStatement(sql);
    // ... 业务逻辑
} catch (SQLException e) {
    e.printStackTrace();  // 仅打印，上层无法感知
    return null;
} finally {
    closeResources(conn, pstmt, rs);  // 重复的资源关闭
}
```

**建议**：使用泛型DAO基类，通过继承消除重复代码。

## 四、安全问题汇总

| 序号 | 安全问题 | 涉及位置 | 严重程度 | 攻击风险 |
|:----:|---------|---------|:--------:|----------|
| 1 | **XSS跨站脚本攻击** | 所有JSP输出处 | 🔴 高 | 攻击者可注入恶意脚本，窃取用户Cookie、会话劫持 |
| 2 | **敏感信息泄露** | `UserDAO.java` 日志 | 🔴 高 | 日志中记录明文密码、加密密钥，一旦日志泄露则全站密码暴露 |
| 3 | **CSRF跨站请求伪造** | 所有表单提交 | 🟡 中 | 攻击者诱导已登录用户执行非预期操作（如修改密码、转账） |
| 4 | **密码加密算法不安全** | `DESUtil.java` | 🟡 中 | DES算法已被破解，存在彩虹表攻击风险，应升级为BCrypt |
| 5 | **SQL注入风险** | 部分动态SQL | 🟢 低 | 目前已使用PreparedStatement，风险较低，但需注意动态拼接的SQL |
