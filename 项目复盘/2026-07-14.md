- **失败路径**：RuntimeException触发回滚、SQLException包装
- **边界**：null返回值、自动提交状态恢复、多步骤部分失败全部回滚
- **重载方法**：executeWithoutResult、executeWithConnection等4种签名

#### 1.2 最小实现（Green阶段）
实现4个公共方法：
```java
// 有返回值，外部传入连接
<T> T execute(Function<Connection, T> action, Connection conn)

// 无返回值，外部传入连接
void executeWithoutResult(Consumer<Connection> action, Connection conn)

// 有返回值，内部管理连接获取/关闭
<T> T executeWithConnection(Function<Connection, T> action)

// 无返回值，内部管理连接
void executeWithoutResultWithConnection(Consumer<Connection> action)
```

核心行为：
- 成功：getAutoCommit → setAutoCommit(false) → 业务执行 → commit → 恢复autoCommit
- RuntimeException：rollback → 恢复autoCommit → 抛出
- SQLException：rollback → 包装为RuntimeException → 抛出

#### 1.3 重构优化（Refactor阶段）
重构要点：
- 将`getAutoCommit()`拆分为`getAutoCommitSafe()`（失败返回保守值true）+ `disableAutoCommit()`（失败抛异常），职责单一
- 提取`rollbackQuietly()`、`restoreAutoCommit()`私有方法，消除重复
- 消除`executeWithoutResult`中的冗余lambda包装

## 二、用到的技术与知识点

### 2.1 Java函数式编程
- **Function<T, R>**：接收参数返回结果，用于`execute`方法
- **Consumer<T>**：接收参数无返回，用于`executeWithoutResult`方法
- **Lambda表达式**：简化事务回调写法
- **@FunctionalInterface**：确保`ConnectionCallback`是函数式接口

### 2.2 JDBC事务管理
- **setAutoCommit(false)**：开启手动提交模式
- **commit()**：提交事务
- **rollback()**：回滚事务
- **getAutoCommit()**：获取原始自动提交状态
- **finally恢复autoCommit**：确保事务结束后状态正确

### 2.3 TDD开发流程
- **Red-Green-Refactor循环**：先写失败测试 → 写最小实现 → 重构优化
- **测试金字塔**：单元测试快速反馈（毫秒级）
- **断言规范**：AssertJ流式assertThat

### 2.4 Git协作
- **git reset HEAD**：重置暂存区，精确控制提交内容
- **git add <文件列表>**：精确暂存所需文件，避免误提交
- **语义化提交信息**：feat/fix/docs等前缀明确变更类型

## 三、遇到的问题与解决

| 问题 | 原因 | 解决 |
|------|------|------|
| Lambda类型推断错误 | `Supplier<T>`不接受Connection参数，无法在lambda中访问conn | 改用`Function<Connection, T>`作为函数式接口 |
| SQLException在lambda中传播 | 函数式接口声明的异常检查与lambda内部SQLException不兼容 | 将所有SQLException包装为RuntimeException，lambda内部不再声明检查异常 |
| `executeWithConnection`中lambda返回void不匹配泛型T | `conn -> { action.accept(conn); }`返回void而非T | 分离为两个方法，内部管理连接的方法直接调用有参版本 |
| 编译错误：类型变量T推断失败 | `executeWithoutResult`内部调用`execute`时lambda返回void无法匹配泛型 | 使用`wrapConsumer`辅助方法显式转换类型 |

## 四、经验总结

1. **函数式接口选择决定API可用性**：Java函数式中`Supplier`不接受输入参数，`Function`接受一个参数返回结果，`Consumer`接受一个参数无返回。根据业务场景选择正确的函数式接口是关键。

2. **最小实现原则**：先让测试通过，再考虑重构。不要在实现阶段添加"将来可能用"的API。

3. **SQLException处理策略**：在lambda内部将检查异常转换为非检查异常（RuntimeException），保证函数式接口的简洁性。

4. **Git协作精确控制**：使用`git reset HEAD`重置暂存区后，用精确的文件列表暂存，避免误提交非相关文件。

5. **测试即文档**：测试用例的方法名`should_[期望]_[when_条件]`格式清晰表达了业务场景，测试代码即是最好的接口文档。
