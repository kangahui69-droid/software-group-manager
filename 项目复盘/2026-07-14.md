## 二、用到的技术与知识点

- **Java函数式编程**：Function<T,R>接收参数返回结果，Consumer<T>接收无返回，Lambda表达式简化回调，@FunctionalInterface确保函数式接口

- **HikariCP连接池**：HikariConfig配置、HikariDataSource单例、HikariPoolMXBean池状态监控、double-check locking懒加载
- **TDD开发流程**：Red-Green-Refactor循环、测试金字塔（单元快/DAO中/集成慢）、AssertJ流式断言
- **配置管理**：双配置文件加载优先级（local覆盖default）、类型安全getter、泛型解析消除重复
- **代码重构**：方法拆分（职责单一）、常量分组、泛型提取消除重复代码
- **Git协作**：git reset HEAD精确控制暂存、git add文件列表避免误提交、语义化提交信息（feat/refactor/docs）、隐私文件通过.gitignore保护

## 三、遇到的问题与解决

| 问题 | 原因 | 解决 |
|------|------|------|
| Lambda类型推断错误，Supplier<T>无法访问Connection | Supplier不接受输入参数，lambda中拿不到conn | 改用Function<Connection, T>作为函数式接口 |
| SQLException在lambda中传播 | 函数式接口声明的异常检查与lambda内部SQLException不兼容 | 所有SQLException包装为RuntimeException，lambda内不声明检查异常 |
| executeWithConnection中lambda返回void不匹配泛型T | Consumer lambda返回void无法匹配Function泛型 | 分离为独立方法，内部直接调用有参版本 |
| 编译错误：类型变量T推断失败 | executeWithoutResult内部调用execute时lambda返回void不匹配泛型 | 使用辅助方法显式转换类型 |
| Result类不存在导致测试编译失败 | TDD流程先写测试后写实现 | 创建临时stub验证语法正确性，验证后删除 |
| HikariCP依赖已在pom.xml但未启用 | DBUtil一直使用DriverManager每次新建连接 | 将getConnection()替换为HikariDataSource池化 |
| getIntProperty方法不存在 | Config类尚未实现该方法 | 临时在DBUtil中实现parseIntProperty，完成后替换为Config.getIntProperty |
| 半成品代码配置key前缀错误（`hikari.`vs`hikaricp.`） | 实现时未对照config.properties模板实际键名 | 修正为`hikaricp.`前缀，与配置文件一致 |
| String类型getter缺少trim/空值处理 | 初版直接调用getProperty返回原值 | 新增parseString()统一处理trim和空值回退 |
| DEFAULT_DB_URL不完整（缺少useUnicode等参数） | 初版常量只写了基础URL | 补全为与原DBUtil内联字符串一致的完整JDBC URL |
| reloadConfig测试断言期望值错误（预期20实际10） | 混淆DEFAULT常量与config.properties模板配置值（maximumPoolSize=10） | 修正断言为10，验证从配置文件加载的值 |
| config.properties模板硬编码真实密码和API Key | 之前只清理了启动说明.md，未检查config.properties | 密码/Key替换为占位符，注释说明使用local配置 |
| NON_KEYWORDS加SET导致所有UPDATE失败 | SET是SQL语法关键字，加入NON_KEYWORDS后H2把SET当列标识符 | 从NON_KEYWORDS移除SET（SQL关键字不需要加） |
| ActivityService test: UnnecessaryStubbing错误 | @BeforeEach setup多个测试共享mock导致部分stub未使用 | 添加@MockitoSettings(strictness = Strictness.LENIENT) |
| ActivityService test: 批量审批测试mock数量不足 | batchApprove调用两次batchUpdateStatus，verify次数不对 | 增加verify次数或改用anyInt()匹配 |
| ActivityService: cancelActivity未调用findByActivityId | 取消活动时未发送通知消息 | 添加findByActivityId调用并发送通知 |
| UserServiceTest: 邮箱唯一性检测异常信息乱码 | 异常信息编码不一致导致contains匹配失败 | 改用更通用的"已被使用"字符串 |
| UserServiceTest: updateProfile的name字段测试数据不匹配 | 测试传入name与mock用户name不一致导致校验失败 | 修正测试数据使name匹配或移除name校验stub |
| UserServiceTest: listMembers需要operatorId参数 | 原实现无权限校验，测试未mock findById | 修改方法签名增加operatorId参数 |
| UserServiceTest: login的密码错误vs用户不存在区分 | findByUsernameAndPassword返回null无法区分两种情况 | 先调用findByUsername检查用户是否存在再验证密码 |

## 四、经验总结

1. **函数式接口选择决定API可用性**：Java中Supplier不接受输入参数，Function接受一个参数返回结果，Consumer接受一个参数无返回，根据业务场景选择正确接口是关键。

2. **TDD Red阶段先验证编译失败再写实现**：先写测试然后运行test-compile确认失败（缺失方法数量与预期一致），确保测试真正定义了接口契约，而不是"测试跟着实现走"。

3. **最小实现原则**：先让测试通过再考虑重构，不要在实现阶段添加"将来可能用"的API。

4. **SQLException处理策略**：在lambda内部将检查异常转换为非检查异常（RuntimeException），保证函数式接口简洁性。

5. **Refactor阶段必须对照实际配置文件检查key名**：不能凭记忆写配置key，必须打开config.properties核对每个key的拼写和前缀。

6. **配置文件模板安全检查是重构的必要环节**：重构Config类时顺便检查config.properties发现了密码泄露问题，主动检查比被动修复成本低。

7. **Git协作精确控制**：使用`git reset HEAD`重置暂存区后用精确文件列表暂存，避免误提交非相关文件；隐私文件通过.gitignore保护。

8. **测试即文档**：测试方法名`should_[期望]_[when_条件]`格式清晰表达业务场景，测试代码是最好的接口文档。

9. **魔法值抽取为常量**：错误码、默认值等应抽取为命名常量，提高可读性和可维护性；常量按业务域分组更易查找。

10. **泛型方法提取消除重复**：parseInteger和parseLong逻辑高度相似，用Function<String,T>提取后代码量减少且逻辑统一，未来扩展只需一行委托。

11. **方法拆分遵循职责单一**：loadConfig拆分为3个方法后，loadLocalConfig处理覆盖加载、loadDefaultConfig处理兜底加载、loadPropertiesFromResource负责IO，每个方法只做一件事，比大方法可读性好得多。

12. **密码类配置不应该trim**：getDbPassword()直接走getProperty不经过parseString，因为密码首尾可能有刻意空格，trim会破坏语义。String配置是否trim要按业务语义判断。

13. **连接池池状态监控很重要**：getPoolStatus()通过HikariPoolMXBean获取真实运行状态，便于线上监控和故障排查。


14. **每次提交后跑全量测试**：重构后跑mvn test全量测试，确保没有破坏依赖模块（DBUtil依赖Config读取HikariCP参数）。

15. **TDD stub阶段验证编译失败数量**：Red阶段写完测试后运行test-compile确认失败数量与预期一致（如58个UnsupportedOperationException），确保测试真正定义了接口契约。

16. **测试数据必须与mock返回值一致**：测试中传入的参数值必须与stub的返回值匹配，否则校验逻辑会错误失败。

17. **方法签名变更需要更新所有调用点**：给listMembers增加operatorId参数后，测试中的所有调用都需要更新传递正确的operatorId。

18. **业务校验顺序影响测试设计**：先检查用户存在再验证密码，可以区分"用户不存在"和"密码错误"两种情况，返回不同错误码。

19. **重构保持测试通过是基本原则**：每次重构后立即运行相关测试，确认行为未改变。

20. **内部类封装减少参数传递**：FileInfo内部类封装文件信息（size/contentType/fileName），减少方法间传递的参数数量。
